## Context

Recurrence today is approval-gated and serial. A recurring task is a single materialized "tip" row; the next instance is created only inside the `POST /:id/approve` transaction in `server/src/routes/tasks.ts`, using `nextDueDate(...)`. Chains link via `templateId`/`instances`. Consequences:

- Only one live instance exists at any time.
- No parent approval ⇒ no next instance, indefinitely.
- A child who misses several days sees one stale overdue row, not the missed occurrences.

The stack is Express + Prisma + SQLite run directly via `tsx` — **there is no cron or background worker**, so any time-driven generation must be triggered by an incoming request. The codebase already has read-time schedule projection precedent: `GET /api/tasks/calendar` calls `projectOccurrences(...)` (bounded by `MAX_STEPS = 400`) to render future occurrences without persisting them.

## Goals / Non-Goals

**Goals:**
- Let a child see and individually complete multiple missed occurrences of a recurring chore without waiting on parent approval.
- Make catch-up an explicit, per-task opt-in so parents who don't want a backlog pile aren't surprised.
- Reuse existing schedule math (`nextDueDate`) and bounding (`MAX_STEPS`); add a time window so a long-ignored daily task can't explode.
- Leave all existing recurrence behavior intact for tasks that don't opt in.

**Non-Goals:**
- Up-for-grabs recurring chores are out of scope — they keep approve-time, pool-reopen spawning.
- Recurring tasks with no `dueDate` (no schedule anchor) are out of scope — no backfill.
- No background/cron materialization; generation is strictly request-triggered.
- No change to per-unit chores.

## Decisions

### 1. Opt-in via a new `catchUp` boolean on `Task`
A new `catchUp Boolean @default(false)` column. Catch-up behavior applies only when `isRecurring && recurrence && catchUp && assignedToId != null && !isUpForGrabs`. Mirror the field through `client/src/types/models.ts` and accept it in create/edit.

*Alternative considered:* global always-on backfill. Rejected — some parents would find a guilt-pile of overdue chores demoralizing, and it would change behavior for every existing recurring task without consent.

### 2. Eager materialization, triggered lazily on read
On `GET /api/tasks`, before returning, backfill missing occurrences for the requester's catch-up templates by creating real `PENDING` `Task` rows. Real rows mean complete (`PUT /:id`), approve, and per-occurrence crediting all work through existing endpoints with no new routes and no synthetic-id special-casing.

*Alternative considered:* virtual occurrences (project on read, materialize only when the child completes one). Rejected — "complete a thing that isn't a row yet" requires parsing synthetic ids and backdating a row across multiple endpoints; more special-casing than it saves, given a window keeps row growth small.

### 3. Backfill algorithm (windowed, bounded)
For each catch-up template visible to the requester:

```
root        = templateId ?? id
lastDue     = max(dueDate) across rows sharing the chain root
windowStart = now − CATCHUP_WINDOW_DAYS          // named constant, default 14
cursor      = lastDue
for i in 0..MAX_STEPS:
    cursor = nextDueDate(cursor, recurrence, weeklyDays)   // reuse existing fn
    if cursor > now: break
    if cursor >= windowStart and no row exists at (root, cursor):
        create PENDING occurrence(dueDate = cursor, templateId = root, catchUp = true, ...)
```

Occurrences older than the window are skipped (the pre-existing stale tip row is left untouched, not deleted). Occurrences carry the same `title/description/dollarAmount/recurrence/weeklyDays/assignedToId/catchUp` as the template.

### 4. Approval credits but does not spawn (for catch-up)
Add a `!task.catchUp` guard to the spawn block in `POST /:id/approve`. Catch-up occurrences are generated by the schedule on read, so approval must not also create a successor — otherwise every approval double-generates. Crediting `dollarAmount` per approved occurrence is unchanged and is the intended payout (7 mail days done = 7× credit).

## Risks / Trade-offs

- **Write-on-read concurrency** → two simultaneous `GET /api/tasks` could both pass the existence check and insert the same `(root, dueDate)` occurrence. SQLite serializes writes but the check-then-insert is not atomic. *Mitigation:* wrap the backfill in a transaction that re-checks existence immediately before each insert; if feasible, add a unique guard on `(templateId, dueDate)` for catch-up rows (note: a partial unique index is awkward in SQLite/Prisma because `templateId` is null for non-instances and shared by per-unit logs — so the transactional re-check is the primary mitigation, with a duplicate-tolerant read as fallback).
- **Side effects inside a GET** → mutating on a read is unusual. *Mitigation:* keep it idempotent and bounded (window + `MAX_STEPS`); document it; it mirrors the read-time projection precedent already in `GET /calendar`.
- **Window default is a product guess** → 14 days may be too short or long. *Mitigation:* define `CATCHUP_WINDOW_DAYS` as a single named constant so it is trivial to tune; no magic numbers inline.
- **Backlog volume for daily chores** → even windowed, a daily catch-up task yields ~14 open rows. *Mitigation:* the window caps it; UI can group/collapse if needed (not required for this change).

## Migration Plan

1. Prisma migration adds `catchUp` with `@default(false)` — backward compatible, all existing tasks default to no catch-up, so behavior is unchanged for them.
2. Deploy follows the standard Railway flow (`prisma migrate deploy` then start). No data backfill needed.
3. Rollback: the column is additive and unused by non-opted-in tasks; reverting code leaves the column harmless.

## Open Questions

- Should the UI group/collapse a daily backlog, or list each overdue occurrence flat? (Out of scope here; flat list is acceptable for v1.)
- Is 14 days the right default window? (Constant makes this easy to revisit.)
